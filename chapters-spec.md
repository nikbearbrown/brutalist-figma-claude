## Chapter Anatomy Template (Practitioner Handbook)

Each chapter follows the failure-first structure:

1. **Opening failure** — a concrete, production-realistic failure mode that the reader has probably encountered or can immediately imagine.
2. **Diagnosis** — what is actually happening, conceptually and in the API response/file structure.
3. **Working solution** — the CLI artifact or technique that resolves the failure, with real code.
4. **Failure modes of the solution** — what can still go wrong, and the decision rules for handling it.
5. **Bridge** — one sentence connecting this chapter's output to the next chapter's problem.

Self-contained: each chapter can be read and applied independently after Chapters 1–7.

## Condensed Per-Chapter Specifications

### Part One — The Gap

**Chapter 1 — Why Your Figma File Is Lying to You**
- **One-line:** The reader learns to recognize the designer-developer gap as a synchronization failure and can state when a pipeline is required vs. when manual export is acceptable.
- **Opening failure:** The designer changes the brand primary color in Figma. Three weeks later production still has the old blue. No error was thrown.
- **CLI artifact:** None in this chapter — conceptual foundation only.
- **Key content:** Synchronization problem vs. communication problem; the Figma file as a document graph; why manual export is inherently a one-time operation; three failure modes (silent drift, version chaos, broken trust); the `tokens_final_v3_really_final.json` anti-pattern; decision rules for when manual is acceptable.
- **Bridge:** Now that we know *why* the pipeline is necessary, Chapter 2 establishes what the API actually exposes to build it on.

**Chapter 2 — What the API Actually Exposes**
- **One-line:** The reader understands the five API surfaces, rate-limit architecture, and authentication, and can run `figma-ping.js` to verify session health before any pipeline work begins.
- **Opening failure:** First API call returns 403. The reader does not know if the problem is the token, the plan, the file permission, or the endpoint.
- **CLI artifact:** `figma-ping.js` — verifies token validity, file key access, plan/seat access, endpoint reachability, and rate-limit headers; prints a clear next action for each failure.
- **Key content:** Five API surfaces (REST, Plugin, Variables, Webhooks, MCP); rate-limit tiers and the Starter-plan trap; PATs vs. OAuth vs. Plan Access Tokens; the Enterprise gate and non-Enterprise workarounds; `FIGMA_TOKEN` and `FIGMA_FILE_KEY` environment contract; `.env` hygiene; failure-first endpoint diagnosis.
- **Bridge:** Now that we can reach the API reliably, Chapter 3 teaches us to read what it returns.

**Chapter 3 — Reading a Figma File Programmatically**
- **One-line:** The reader can navigate the raw Figma document graph, write a local fixture for offline testing, and extract a component inventory with `figma-read.mjs`.
- **Opening failure:** `GET /v1/files/:key` returns 12,000 lines of nested JSON. The reader has no idea where their components are.
- **CLI artifact:** `figma-read.mjs` — fetches the file graph, normalizes the response, writes a stable local fixture, prints file/page/component inventory.
- **Key content:** The node tree (document → canvas → page → frame → component → layer); variables, collections, modes, alias chains; local vs. published components and styles; what is NOT in the response (prototype logic, font files, interaction states); fixture-based testing rationale; output shape (inventory JSON, page/frame report, missing-metadata report).
- **Bridge:** We can now read any Figma file. Chapter 4 establishes the naming discipline that determines whether what we read is useful.

## CLI Operating Spine

The CLI artifacts introduced across the book form a stable, composable toolkit:

| Artifact | Chapter | Purpose |
|---|---|---|
| `figma-ping.js` | 2 | Session health check before any pipeline work |
| `figma-read.mjs` | 3 | Read file graph, write fixture, print inventory |
| `figma-audit.js` | 5 | Programmatic audit (naming, brand, WCAG, token hygiene) |
| `figma-fix-plugin/` | 6 | Staged Plugin API remediation with human approval |
| `FIGMA.md` (pipeline) | 7 | Governing file: pipeline authorization declaration |
| `extract-tokens.mjs` | 8 | Variables API extraction + DTCG output |
| `validate-tokens.mjs` | 8 | Pre-transform token validation |
| `export-assets.mjs` | 9 | Batched asset export with SVG post-processing |
| `sync-docs.mjs` | 10 | Component inventory and documentation sync |
| `monitor-brand.mjs` | 11 | Programmatic brand and WCAG compliance report |
| `build-spec.mjs` | 12 | Machine-readable component specification JSON |
| `FIGMA.md` (agent) | 13 | Agent governance file for MCP sessions |

**Stable CLI contract taught throughout:**
1. Environment: `FIGMA_TOKEN`, `FIGMA_FILE_KEY`, no hardcoded secrets, local `.env` hygiene.
2. Input: file key or node list, plus a declared consumer (human report, token compiler, asset pipeline, code generator, AI agent).
3. Output: human-readable markdown AND machine-readable JSON where downstream automation is expected.
4. Validation: schema checks, rate-limit handling, retry/backoff, fixture-based tests, explicit failure codes for CI.
5. Governance: every command states what it reads, what it writes, what it can automate, and what requires human review.
